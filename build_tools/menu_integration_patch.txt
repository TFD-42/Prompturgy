CLI PATH NOTES:
---------------
In CLI mode (main()), there is no collect_input() call — the task comes from
args.task directly. The pre-processor should be applied there too, as a
standalone call before generating.

Pattern in main():

    task = sanitize_input(args.task, "text")

    if settings.get("use_pre_processor", True) and not getattr(args, "no_preprocess", False):
        task = pre_process_input(
            raw_input=task,
            model=settings.get("pre_processor_model") or settings.get("model_a", DEFAULT_MODEL_A),
            ollama_url=args.ollama_url,
            timeout=PRE_PROCESSOR_TIMEOUT,
            stream_callback=None,  # CLI mode: no streaming for pre-processor
        )
        # Optionally print the restructured prompt so user can see what was sent:
        if not args.quiet if hasattr(args, "quiet") else True:
            print(f"[pre-processor] → {task[:200]}{'...' if len(task) > 200 else ''}")


SYNTHESIS PATH:
---------------
menu_synthesis() lets the user pick 2 existing output files and merge them.
The task input here is a short description, NOT the original raw prompt.
Pre-processor is NOT appropriate here — skip it for synthesis.
The prompt in menu_synthesis() is: input("  Original task (short description):\n  > ")
Leave this unchanged.


INTERACTION WITH QUICK MODE:
-----------------------------
Pre-processor runs BEFORE mode selection matters — it always fires first.
Whether the pipeline is QUICK or FULL, the restructured prompt is the input.
This is correct: the pre-processor improves the raw signal regardless of output mode.
